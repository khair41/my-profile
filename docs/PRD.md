# Product Requirements Document

## Overview

A personal profile website that doubles as a live AI intelligence engine. The site showcases the owner's work and perspective while surfacing a continuously-updated digest of dev/AI news, startup ideas, and learning paths — generated by a local Python + Ollama pipeline and curated through a private studio interface.

**Primary audience:** Recruiters, collaborators, and developers who discover the owner via search, GitHub, or referral.

**Secondary audience:** The owner, using `/studio` to review and approve pipeline-generated content.

---

## Phase 1: Next.js Scaffold + Public Profile

### 1.1 Project Scaffold

**What it does:** A new visitor can navigate to the deployed URL and see a functional site. All top-level routes respond with valid HTML. The site loads in dark mode by default. A navigation bar and footer are present on every page.

**Acceptance criteria:**
- Site deploys to Vercel from the main branch with no manual steps beyond initial project creation
- Public routes (`/`, `/news`, `/learning`) return HTTP 200; `/ideas` and `/studio` redirect to `/` on Vercel
- Dark mode is the default; no flash of unstyled light-mode content on load
- Navigation links are present on every page and visually indicate the active route
- Site passes Lighthouse accessibility score ≥ 90 on both mobile and desktop
- Zero TypeScript errors; zero ESLint errors

### 1.2 Profile / Bio Page (`/`)

**What it does:** A visitor sees the owner's name, tagline, professional photo or avatar, and a 2–3 paragraph bio. Below the bio are the owner's tech stack icons and social/contact links. The page communicates what the owner builds and how to reach them.

**Acceptance criteria:**
- Hero section renders name, tagline, avatar, and at least one primary CTA link
- Bio section renders without layout shift on a 375 px viewport
- Tech stack renders as an icon grid; every icon has accessible alt text
- Social links open in a new tab and include at minimum GitHub and email
- All content is editable via configuration constants — no hard-coded strings in JSX
- Page is fully statically generated (no server-side data fetching at runtime)

### 1.3 Placeholder Content Pages

**What it does:** `/news` and `/learning` each render a recognizable page with a heading and an empty-state message until real data is available. `/ideas` is private and does not need a public placeholder.

**Acceptance criteria:**
- Each public placeholder page shares the root layout (nav, footer)
- An empty-state message is shown when the corresponding JSON file contains an empty array
- No JavaScript errors appear in the browser console on any placeholder page

## Phase 2: Python Crawler

### 2.1 RSS Feed Crawler

**What it does:** Running the pipeline fetches the latest entries from all configured RSS/Atom feeds (Hacker News, dev.to, ArXiv, and others). Each entry is saved as a raw article record. Articles already in the knowledge base are skipped. The crawler continues past individual feed failures without crashing.

**Acceptance criteria:**
- Fetches from all RSS sources defined in `sources/config.yaml`
- Deduplicates by URL — re-running the crawler does not produce duplicate records
- Raw article records include: URL, title, plain-text body (HTML stripped), source name, fetched timestamp
- A single unavailable feed is logged as an error and skipped; the rest of the run continues
- Per-request delay is configurable (default 1 s)

### 2.2 GitHub Trending Crawler

**What it does:** The crawler fetches the current GitHub Trending page and records each trending repository. Each record captures the repo name, description, primary language, star count, and URL.

**Acceptance criteria:**
- Captures at least the top 25 trending repositories per run
- Deduplicates by repository URL within the same day's run
- Repositories with no description or missing language field are handled gracefully (empty string defaults)
- No GitHub API token is required (public page); an optional token raises rate limits

### 2.3 YouTube Transcript Crawler

**What it does:** For each configured YouTube channel, the crawler retrieves transcripts of the most recent N videos (default 3). The transcript text becomes the raw article body. Videos with no available captions are skipped with a log warning.

**Acceptance criteria:**
- Fetches transcripts using `yt-dlp` auto-caption extraction for each configured channel
- Videos with no available captions are skipped without crashing
- Transcripts longer than 6000 characters are truncated to fit the Ollama context window
- Deduplicates by video URL

### 2.4 Manual URL Crawler

**What it does:** A YAML file at `sources/manual.yaml` lists specific URLs (with optional titles). The crawler fetches and extracts the main article body from each URL using a readability heuristic, enabling one-off inclusion of articles outside configured feeds.

**Acceptance criteria:**
- Reads all entries from `sources/manual.yaml`
- Extracts article body text with boilerplate, navigation, and ads stripped
- Checks `robots.txt` for each domain; skips disallowed URLs with a log warning
- An empty or missing `sources/manual.yaml` produces no error (treated as an empty list)

### 2.5 Crawl Runner

**What it does:** A single CLI command runs all enabled crawlers in sequence and appends new raw articles to the knowledge base. A summary is printed at the end showing how many articles were added vs. skipped.

**Acceptance criteria:**
- `python pipeline/run.py --crawl` runs all crawler modules
- Per-source counts and a final total are printed
- Exits with code 0 on success, non-zero on an unrecoverable error
- Does not overwrite existing knowledge base items — only appends

## Phase 3: Ollama Digest Pipeline

### 3.1 Article Summarization

**What it does:** Running the digest stage processes all unprocessed raw articles in the knowledge base. For each article, Ollama generates a 2–4 sentence summary, a relevance score (0–1), and 3–6 topic tags. The result is written back into the knowledge base record.

**Acceptance criteria:**
- Processes only articles with `status === "pending_review"` (already-processed articles are skipped)
- Each processed record contains: `summary` (string), `tags` (string[]), `relevance_score` (number 0–1)
- On Ollama JSON parse failure, retries once before marking the item `status: "error"`
- Progress is printed as it runs (e.g. "Digested 12 / 15 articles")
- Ollama is not called if there are no new articles to process
- Runs entirely offline; requires only a local Ollama instance at the configured URL

### 3.2 Knowledge Base File

**What it does:** `data/knowledge-base.json` is the persistent source of truth for all pipeline-processed content. It is valid JSON at all times, survives interrupted pipeline runs, and never permanently deletes items (only their status changes).

**Acceptance criteria:**
- File is an object with a top-level `items` array and three `pending_*` arrays
- File is written atomically (write to temp, then rename) to prevent corruption
- Concurrent pipeline runs do not corrupt the file
- Schema validation runs on load; malformed items are logged and skipped, not crash-inducing

### 3.3 Digest Runner

**What it does:** A single CLI flag runs the digest stage against the current knowledge base. It can also be chained with the crawl stage in one command.

**Acceptance criteria:**
- `--digest` alone processes any unprocessed articles already in the knowledge base
- `--crawl --digest` runs both stages sequentially
- Final output shows the count of newly summarized articles

## Phase 4: Generation Modes

### 4.1 News Digest Generation

**What it does:** The news generation mode reads the knowledge base and produces a set of news digest items suitable for display on `/news`. Each item has a title, 1–2 sentence summary, source attribution, and tags. Generated items are staged in the knowledge base for studio review — they are not written directly to `data/news.json`.

**Acceptance criteria:**
- Generates 5–15 news items per run (configurable maximum)
- Each item references the knowledge base article(s) it was derived from
- Items are staged in `pending_news[]` inside `knowledge-base.json`, not in `news.json`
- Re-running generation does not duplicate already-pending items
- `/news` displays items from `data/news.json` sorted by `published_at` descending

### 4.2 Startup Idea Generation

**What it does:** The ideas generation mode synthesises startup ideas inspired by patterns in recent knowledge base articles. Each idea has a title, problem statement, opportunity description, suggested tech stack, and tags. Ideas are staged for studio review. The `/ideas` page is private — accessible only on the local dev server — and `data/ideas.json` is never committed to git or deployed to Vercel.

**Acceptance criteria:**
- Generates 3–8 idea items per run (configurable maximum)
- Each idea includes: `title`, `problem`, `opportunity`, `suggested_stack[]`, `tags[]`
- Items are staged in `pending_ideas[]` inside `knowledge-base.json`
- `/ideas` is gated by `IDEAS_ENABLED=true`; absent on Vercel, it redirects to `/`
- `data/ideas.json` is listed in `.vercelignore` and excluded from the Vercel build
- `/ideas` displays items from `data/ideas.json` when accessed locally
- Generated ideas are meaningfully connected to recent knowledge base content (not generic)

### 4.3 Learning Path Generation

**What it does:** The learning generation mode produces structured learning path suggestions based on topics that appear frequently or with high relevance in the knowledge base. Each path includes a topic, rationale, difficulty level, estimated time, and specific resources drawn from ingested articles.

**Acceptance criteria:**
- Generates 3–6 learning paths per run (configurable maximum)
- Each path includes: `topic`, `rationale`, `difficulty` (beginner/intermediate/advanced), `estimated_hours`, `resources[]`
- Resources reference real URLs from ingested articles where possible
- Items are staged in `pending_learning[]` inside `knowledge-base.json`
- `/learning` displays items from `data/learning.json`

### 4.4 Generate Runner

**What it does:** A CLI flag runs one or all generation modes.

**Acceptance criteria:**
- `--generate news`, `--generate ideas`, `--generate learning`, and `--generate all` are all supported
- `--generate all` runs the three modes in sequence: news → ideas → learning
- An error in one mode does not abort the others when running `all`
- Item counts generated per mode are printed on completion

## Phase 5: Local Studio Review UI

### 5.1 Private Route Access Gates

**What it does:** `/studio` and `/ideas` are only accessible when their respective environment variables (`STUDIO_ENABLED=true`, `IDEAS_ENABLED=true`) are set. On Vercel neither variable is present, so requests to either route immediately redirect to `/`. No login or authentication is required — the gate is the environment variable.

**Acceptance criteria:**
- `STUDIO_ENABLED !== 'true'` causes an immediate redirect to `/` for all `/studio` requests, at the layout level
- `IDEAS_ENABLED !== 'true'` causes an immediate redirect to `/` for all `/ideas` requests, at the layout level
- No studio or ideas content is reachable on the Vercel deployment
- No sensitive data (knowledge-base contents, pending items, unpublished ideas) is ever rendered in public routes
- `data/ideas.json` is excluded from the Vercel build via `.vercelignore`
- Navigation on the public site does not include a link to `/ideas`; the link appears only when `IDEAS_ENABLED=true`

### 5.2 Review Queue

**What it does:** The studio home screen shows all knowledge base items with `status === "pending_review"`. Each item displays its title, source, date, Ollama summary, relevance score, and tags. The reviewer can mark each item as processed (kept in KB, eligible for generation) or rejected (excluded from all future generation).

**Acceptance criteria:**
- Queue loads all pending items sorted by relevance score descending
- Each item displays all `KnowledgeBaseItem` fields
- Approving sets `status: "processed"` and removes the item from the queue view
- Rejecting sets `status: "rejected"` and removes the item from the queue view
- Changes persist to `data/knowledge-base.json` immediately (no separate save step)
- An empty-queue state is displayed when no items are pending

### 5.3 Generation Trigger

**What it does:** Within the studio, the reviewer can trigger any generation mode with a button click. The UI shows a loading state while the pipeline runs and displays the number of newly staged items when complete.

**Acceptance criteria:**
- Buttons exist for each generation mode and for "all"
- Buttons are disabled and show a spinner while the subprocess runs
- On completion: "Generated N items" is displayed, or an error message if the run failed
- The current studio view is not navigated away from during generation

### 5.4 Approval Buckets

**What it does:** The studio has three tabs — News, Ideas, Learning — each showing the pending generated items for that type. The reviewer can approve (writes to the corresponding `data/*.json`) or reject (removes from pending in KB) each item.

**Acceptance criteria:**
- News tab shows all items in `pending_news[]` from `knowledge-base.json`
- Ideas tab shows all items in `pending_ideas[]`
- Learning tab shows all items in `pending_learning[]`
- Approving a news item appends it to `data/news.json` and removes it from `pending_news[]`
- Approving an idea appends it to `data/ideas.json` and removes it from `pending_ideas[]`
- Approving a learning path appends it to `data/learning.json` and removes it from `pending_learning[]`
- All file writes are atomic
- Tab badge counts reflect the current pending item count

### 5.5 Studio Design

**What it does:** The studio UI uses the same Tailwind/Geist design system as the public site but has a distinct visual treatment so the reviewer always knows they are in the studio context.

**Acceptance criteria:**
- Studio layout is visually distinct from public pages (clear "Studio" label always visible)
- All interactive elements have clear hover and focus states
- Studio is usable at a 1280 px viewport (mobile responsiveness not required)
- Keyboard accessible: Tab through items, Enter to approve, Delete/Backspace to reject

## Phase 6: Polish

### 6.1 Animations

**What it does:** Page transitions and card appearances use subtle motion. Cards animate in on first render with a staggered fade-up. Hover states on cards have smooth transitions. Users who prefer reduced motion see no non-essential animation.

**Acceptance criteria:**
- Card grids use a staggered entrance animation
- `prefers-reduced-motion` disables all non-essential animation
- No layout shift is caused by animations
- All animations complete in ≤ 300 ms

### 6.2 Tag Filtering

**What it does:** On `/news`, `/learning`, and locally on `/ideas`, a filter bar shows all tags present in the current data set. Clicking a tag filters the displayed cards to only those matching that tag. Multiple tags can be selected (OR logic). Selecting no tags shows all items.

**Acceptance criteria:**
- A filter bar renders above the card grid on all applicable content pages
- Tags are deduplicated and sorted alphabetically
- Clicking an active tag deselects it
- Filter state is reflected in the URL query string (e.g. `?tags=llm,rag`) for shareability
- Filtering is client-side — no network requests

### 6.3 Search

**What it does:** A search input on `/news`, `/learning`, and locally on `/ideas` filters displayed cards by matching the query against title, summary, and tags. Search is case-insensitive and updates as the user types.

**Acceptance criteria:**
- A search input is visible above the card grid alongside the filter bar
- Matches against title, summary, and all tags (case-insensitive substring)
- Results update with renders completing in ≤ 50 ms for datasets up to 500 items
- An empty search string shows all items
- A "No results" empty state is shown when search + filter yields zero items

### 6.4 Projects Page (`/projects`)

**What it does:** A new public page showcases the owner's notable projects. Each project has a name, description, link (GitHub or live URL), tech stack tags, and an optional screenshot. Content is managed via a static TypeScript config file.

**Acceptance criteria:**
- Route `/projects` is added to navigation
- Project data is defined in `lib/projects.ts` as a typed array (no `any` types)
- Each project card links to its repo or live URL
- Tech stack tags use the same `Badge` component used elsewhere
- Page is statically generated
